已有本地仓库

  1. 在你使用的代码托管平台创建一个远程仓库
  2. git remote add 地址别名 远程仓库地址
  3. git push 地址别名 master

Pull 和 Push

在每次 push 的时候，如果本地仓库的版本和远程仓库版本不一致了，git 是不允许 push 的。

所以，这个时候，就必须在 push 之前先 pull 一下， 拉取最新版本（先确保把本地的版本和远程仓库的版本保持一致）。pull 下来之后，文件与文件之间有的会自动合并，有的会产生冲突（git 也不做这件事儿），所以你找到冲突的文件，找到提交和谁提交的这个版本的文件冲突了的人，两个人一起对比这个文件进行合并。手动合并之后再 commit 这个合并之后的版本，然后再 push。

使用 Git 结合 GitHub 多人协作

Pull Request

专业一点应该具有 Code Review 这个环节，Pull Request 不仅能帮你实现协同，更多的是帮你把 Code Review 的这个环节变的更加方便了。

  • 项目仓库 lipengzhou/github-flow-demo

其他成员如何协同：

  1. 把源代码仓库 fork 到自己的账户下
  2. 从自己的账户下的仓库 clone 到自己的本地
  3. 新建分支（功能、bug、性能修补）
  4. 基于新分支编写你的代码
  5. 在你的本地新分支功能开发完毕之后，把该分支 push 到自己 fork 的那个仓库中
  6. 发起 Pull Request 请求，请求把我的仓库中的这个分支合并到你的某个分支中

然后仓库的拥有管理者：

  • 收到 Pull Request 合并请求
  • 处理 Pull Request 合并请求
    • 看一下对方代码是否符合规范，是否有不合理的地方，是否有改错的地方
    • 如果一旦有问题，GitHub 就像普通的聊天工具一样，告诉对方你哪儿有问题
    • 对方也会收到你的合并反馈
    • 源码提交者和源码管理者就可以以这种方式进行协同交互
    • code review
  • 直到最后源码管理者认为你的代码没有问题，将你提交的代码合并到了我的主仓库中

Collaborators

  • 项目管理者对协同者发出协同邀请
  • 协同者会收到一封协同邀请邮件，在邮件中同意成为该项目的协同者

只要根据上面的操作就可以让一个项目具有多个协同者，提交的时候，所有的协同者都具有直接 push 的权限。
